## Required in-engine testing

A Ren'Py executable and the complete Project Tac runtime were not available in the packaging environment. Before relying on the build, run these checks inside the target project:

1. Open and close Studio from dialogue, exploration, menu, and custom-screen states.
2. Move, resize, rotate, delete, undo, and redo Scene objects.
3. Confirm Scene edits do not refresh or damage captured UI.
4. Switch to UI mode and inspect HUD, location navigation, Quick Menu, and temporary overlays.
5. Confirm inactive developer/debug screens do not appear by default.
6. Edit a source-backed UI widget and confirm only the supported property changes.
7. Confirm parent-controlled VBox/HBox/Grid children report their layout restriction.
8. Switch Frames and verify selection, hierarchy, Properties, Layers, and canvas update immediately.
9. Type into Properties and Asset search, then save/export without changing focus first.
10. Test autosave, forced shutdown, recovery restore, recovery discard, manual snapshots, and snapshot restore.
11. Generate a Project Tac writer plan, inspect its diff, modify the source externally, and confirm stale-plan rejection.
12. Apply a valid single-file plan and verify backup, atomic replacement, hash, and export-history records.
13. Apply an intentionally failing multi-file transaction and verify rollback.
14. Save, reload, continue gameplay, rollback, and load a Ren'Py save after closing Studio.
15. Run Ren'Py Lint on the complete project after any generated source is applied.

## Known risk areas

- custom Python displayables;
- deeply nested transforms and clipped containers;
- repeated screen instances without stable widget IDs;
- ATL and dynamic screen-language loops;
- transclusion and custom statements;
- source generated by unusual formatting conventions;
- nightly Ren'Py internal API changes.

Use **Debugger → Copy Full Report** when reporting a runtime failure.
